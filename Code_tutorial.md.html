NirsPredict code tutorial

Structure

NirsPredict is a R shiny application which means it have a structure splitted in two main parts, a user interface (ui) and an R session that runs code and returns results (server).
There is also a global.R file, not mandatory for a shiny app but it helps reduce redundant code and increase apps speed, we use it here to load libraries and sourcing some list.
There are other R files used in the app but there are specific to a function. Concerning folders, www contains all the images and csv all the datafile used by the app. The folders contribution and uploads should be empty as they are only used to store temporary files.

How it works ?

When a user open the application it will create a unique temporary repository so each user has his personal session. (l4)
This session is represented by a token : session$token which is the name of the temporary repository.
When the user close the apps window, the temporary repository is deleted after 1 hour. (l809)

So that the app can be used at the same time by multiple users, it has been designed in an asynchronous way. Indeed the packages "future"and promise allow to run long process in a separate session preventing it from slowing the app. These process are included in future({}).

The Explore database part of the app is based on the interaction with a postgreSQL database stored on the SIE server, this database contains all spectra and traits values data which a user can consult and extract. A large part of the code is focused on the way to build dynamic queries to interact with this database at the users will.

On the other hand the Predict trait part is based on the interaction with a computer equipped with a Gpu whos running the python deep-leraning scripts which make the predictions. The interaction is possible thanks to a remote control of this computer thourgh a ssh process. The code allow to transfer files between the app folder and the computer, execute process, interact with the python files and create temporary folders.

Interface (ui)

The interface is composed of a main container :

navbarPage(title = "NirsPredict", id = "tabset"

itself composed of 3 panel tabPanel representing each page of the app Homepage / Application / About , the same applies to the Application panel which is composed of 3 panel representing the 3 subpart Predict trait / Explore database / Become contributor.
In some of these panels we use fluidrow()to control a fixed layout of the page. This layout is horizontally composed of 12 units, then to divide the page in two columns, columns(width = 6) is used. Using two of these formula we can have 2 inputs appearing side to side in columns of equal size on the app page.

It is fundamental to know that almost each element of ui.R include an id to refer to and a name under which this element will appear on the page. The id allow to refer to this element and interact with it in server.R

pickerInput("plant_stage",'Plant Stage',multiple=TRUE,choices=NULL,options = list(`actions-box` = TRUE)))

For example in the above input, the id is plant_stage while the name under which this input will appear to the user is Plant Stage.

Moreover the package shinyjs is used here to hide or disable some input/button, to do so on not only an input but several ones at the same time we have to encapsulate these elements in a div. We can then in server.ui show / enable / toggle these elements with the id of the div.

Server

The file server.R has been annotated as much as possible to associate each part of the code to a clear function and make the code clearer.

CREDENTIALS(6-12){\textsf{\color{blue}CREDENTIALS (6-12)}}CREDENTIALS(6-12)
Credentials required to access the database and the Gpu are stored in the file credentials.csv, this file is read and each of these elements are stored in a variable.

CONSULTATIONFILTERSUPDATE(19-41){\textsf{\color{blue}CONSULTATION FILTERS UPDATE (19-41)}}CONSULTATIONFILTERSUPDATE(19-41)
The pickerInput on the ui.R are dynamically based on the database so they do not contains any value at start. Thats why we have to query the database to retrieve each unique values of each parameter.

assign(paste("SqlOutput",i,sep=""),dbGetQuery(con, query)

The above code dynamically assign the result of the query to a variable composed of SqlOutput + the parameter name.
Then the pickerinput corresponding to each parameter are updated with the values contained in the corresponding variable.
We have to open the connection to the database before the whole process with dbconnect() then close the connection once the process is done with dbDisconnect().

DATABASEQUERYMANAGEMENT(44-178){\textsf{\color{blue}DATABASE QUERY MANAGEMENT (44-178)}}DATABASEQUERYMANAGEMENT(44-178)
The dbManagement function allow to dynamically build query according to the filters selected by the user.
First, two lists are declared, one containing all the parameters inputs and one for their corresponding id. We go through the input list to check which one is filled and then concatenate the corresponding parameter names (45-58).
Then for each parameter we gather all values entered by the user in an other character list. This list paramList is then incorporated into the syntax of a dynamic query which is stored in another list filterList1 containing all queries corresponding to each parameter (61-72).
The content of the parameter list filterList and the queries list filterList1 are put together and edited to match a correct postgreSQL syntax (74-79).

SPECIAL CASES
To deal with inputs whos not a list of characters some special rules and conditional cases has been set. To treat with inputs like the date or radio buttons inputs a new list filterListFinal has been built which is the concatenation of the previous queries list and these special cases queries (82-116).

OUTPUT FORMAT
As its possible to choose the query result output format, we have to adapt the query to the users choice. There is some base parameters whos retrieved every time stored in basicParameters variable. For the custom output format there are 4 additional inputs which are scanned through a loop to store the corresponding value in a list OtherFilterFinal (119-133). If the list contains values then they are concatenante to the list of parameters to retrieve, else only the base parameters are retrieved (134-140).

According to the output format selected by the user and if some filters have been chosen or not we are able to build different query, a query for each case is built and stored in a list (142-178).

FORMMANAGER(181-212){\textsf{\color{blue}FORM MANAGER (181-212)}}FORMMANAGER(181-212)
This part of the code manage all the processes triggered when the user submit his form to explore the database content.
A connection to the database is opened the previously seen dbManagement function is executed and the resulting queries are stored in variables, then the spectra query is executed and the connection closed (186-194).

The following processes are executed in an asynchronous way because they can take some time according to the number of values retrieved from the database.
The function FormatData is used to transform the raw results of the spectra query into user readable table. The function outputManagement manages the writing of the results according to the output format chosen by the user.
The functions meanPlot() and pcaSelectedPlot() are designed to print a graphical analysis of the results.

WRITINGCSVOUTPUT(213-294){\textsf{\color{blue}WRITING CSV OUTPUT (213-294)}}WRITINGCSVOUTPUT(213-294)
A connection to the database is opened and the query to get traits values is executed, the result paramsOnlyRes are scanned to hide the genotype if its an unpublished data (217-224).A Csv result file is written for the traits values and another for the spectra.
For each ouput format selected, the result of the query is checked to see if its not empty. If the All Data format output is selected and the result is not null, then trait values table and spectra table are merged and written as csv file. The donwload button is then showed and the file uploaded (228-243). The process is roughly the same for each output format.

MEANPLOTMETHOD(323-363){\textsf{\color{blue}MEAN PLOT METHOD(323-363)}}MEANPLOTMETHOD(323-363)
This function aims to compare the spectra values contained in the sample selected by the user and the spectra values of the whole database in a mean comparaison. To do so, a new dataframe is built containing the mean value of arbsorbance for the selected spectra from 350 to 2500 (325-334). A dataframe containing mean value of all databases spectra is stored in the csv folder. These two dataframe are merged and edited to be plotable (336-345). A solid line is used to represent the selected sample while the database values are represented with a dotted line.

PCA(367-380){\textsf{\color{blue}PCA (367-380)}}PCA(367-380)
This function aims to produce a PCA of the spectra values contained in the sample selected by the user. It allows to visually compare the graphical result with the Pca of all databases spectra values. It is important to know that the Pca plot is stored as an image in the www folder.

DENSITYGRAPHICCOMPARISON(384-401){\textsf{\color{blue}DENSITY GRAPHIC COMPARISON (384-401)}}DENSITYGRAPHICCOMPARISON(384-401)
This function aims to compare a chosen trait predicted values against the database values for this trait. First, a query to retrieve all trait corresponding values is sent to the database (386-390). Then the prediction file is read from the user temporary session folder and we plot the density of those two list of values (392-400).

FILEUPLOADCHECKING(425-477){\textsf{\color{blue}FILE UPLOAD CHECKING (425-477)}}FILEUPLOADCHECKING(425-477)
globalUploadCheck() is the function called in the main process to check is the uploaded files format are correct. This function is splitted in two sub functions, one for the spectra file and another for the traits file. The function choose which sub functions should be used according to the chosen option and return TRUE if files format are correct (409-423).
traitUploadCheck() starts by checking if a file has been provided otherwise an alert will pop. Then the file is read and we check if it contains NA, in this case the uploading input is reset. Finally the presence of the destination folder (uploads) is checked and the file is downloaded in there (425-450).
Like before spectrumUploadCheck() check the presence of the file and the presence of NA otherwise an alert will pop. Moreover the files columns number is checked because there must be at least 400 values of absorbance by spectra to use the prediction algorithms (457-477).

GPUINTERFACE/RUNNINGPYTONPIPELINE(480-628){\textsf{\color{blue}GPU INTERFACE / RUNNING PYTON PIPELINE (480-628)}}GPUINTERFACE/RUNNINGPYTONPIPELINE(480-628)
This is the main process for predicting trait values. First of all the globalUploadCheck() function is called to to check if the provided files format are correct. The process is divided in three parts, one for each mode of operation. Because it can take some time the process is ran in an asynchronous way (using future()). Moreover each process is included in a tryCatch() to handle error and preventing the app to crash if something go wrong.
The ssh package is used here to do remote control to interact with the computer equipped with a GPU.

ssh_connect(ipGpu, passwd = passwordGpu)

The previous command allow to create an ssh session which can be seen as a bridge between the app and the Gpu computer, the IP adress and the password of this computer is needed to create the session.

ssh_exec_wait()
ssh_exec_internal()

The two previous command allow to execute a bash command on the opened session (on the computer equipped with a Gpu in our case). To do so, we need to provide the session name and the command to be executed under the form of a character variable.

scp_upload(sessionGpu,file.path,to=paste("/home/vaillant/Documents/pyNirs/",session$token,sep = ""))

The previous command allow to upload a file from the app folder into the computer equipped with a Gpu. It must be provided the location of the file and the location in the remote computer where it should be stored. In our case the app folder on the remote computer is named pyNirs and a temporary folder is created in it with the session token. name.
On the opposite the scp_download() command allows to download a file from the remote computer into the chosen folder (the users one here).

When a csv file is uploaded into the remote computer it is automatically renamed as 0.csv so we have to rename it as the generic name Xcal.csv so it can be detected by the python scripts (510).
The trait input is scanned so a list of every chosen traits can be build (512-515).

outBash<-ssh_exec_internal(sessionGpu,paste("bash /home/vaillant/Documents/pyNirs/setup.sh 1 ",session$token,listTraits,sep=""))

The previous command aims to launch the setup file on the remote computer. In this command we provide a number following the setup file name which represents the running option chosen by the user, we also provide the names of the chosen traits. This file will run the python scripts according to the chosen running options. The results of the command is stored in a variable (outBash) so we can check if there is no error in the run otherwise the process is stopped and an alert pop.
Once the scripts are done running, we retrieve the result folder into the users session folder and the temporary folder on the remote computer is deleted. As there is one csv file of predictions for each chosen traits, we execute the DensityComparison() function for each one (525-529).
Finally the sendResults.R file is launched with the registered mail provided by the user as argument, so the script can automatically send the result files to the user by email. This file will be described below.

The code is almost the same for the 3 options, the number provided as argument to the execution of the setup file is different according to the chosen option.
For the second option (Predict traits with your own model) the trait values file is also uploaded on the remote computer and renamed as Ycal.csv (554-555).
For the third option (Test your model) the test dataset is also uploaded on the remote computer. The test spectra file is renamed as Xval.csv and the test trait values file as Yval.csv (598-601).

Automatic mailing system

Two files are used to send mail, sendResults.R for the emails related to the Predict trait page and newContribution.R for the emails related to the Become contributor page.
This system is based on the use of a third-party platform : Mailjet. Mailjet is a cloud-based email delivery and tracking system which allows users to send emails. Here is a link to the Mailjet guide : https://dev.mailjet.com/email/guides/ , the SEND API V3 is used here.

We mostly use the package httr in those files, it provide a wrapper and web requests customised to the demands of modern web APIs like GET() or POST().

sendResults.R{\textsf{\color{blue}sendResults.R}}sendResults.R
The first line in the following code is here so the file can be detected as a R script by the bash system.
The second line is here to retrieve all arguments provided with the launch command of this file.

#!/usr/bin/env Rscript
args = commandArgs(trailingOnly=TRUE)

To use the Mailjet API we have an id (api_key) and a password (api_secret) specific to our account, they are stored in the credentials.csv file. We also have a specific email address from which the email are sent : [email protected].
All files in the Res folder located in users session folder are listed and assign to a variable (17).
Those files are concatenated into a folder named Results and then under the form of a zip file named Results.zip (18-26).
A POST request is then build to be sent to the Mailjet API. The url of the API is provided (30) as the api_key and api_secret (31-34) which are required to be authenticated with the API.
The content of the email itself is under the JSON format using the jsonlite package.
We fill in the From field with the sender email and the To field with the users email (42-50).
The title of the mail is filled in line 52, then the body as normal text but also in HTML (53-64). We also join the attached files as zip content with the name under which it will appear (65-70).
Once the mail is sent, bash commands are used to delete the results folders (78-79).

Remote computer side (Python)

As previously mentionned, the python scripts which allows to get predictions are located on a remote computer equipped with a Gpu. The application folder on this computer is /home/vaillant/Documents/pyNirs. It contains several folders and files. The models folder contains an estimator for each of the 81 traits covered by the application, they are mandatory to make predictions from spectral data. The weights folder also contains weight related to each of these etimator. Together the estimator and the weight related to a trait form the Built-in models mentionned in the app.
The Scripts folder contains the generalPipeline.py which is the python pipeline.
There is also two log files, std.log which is the actual log file and std.log.1 which is the archived log file.
The bash file setup.sh is here to ensure the link between the command given by the Shiny application and the python script.

Setupfile{\textsf{\color{blue}Setup file}}Setupfile
This script get the arguments provided by the shiny application and transmit them in a bash syntax to launch the python pipeline. The first line #! /bin/sh is here to declare the file content as bash. The list of traits provided as an argument is scanned and stored in a bash list (11-17). According to the option chosen by the user, the python script is launched with as argument the option number,the session name and the list of traits if first option selected.
